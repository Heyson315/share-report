Severity-Based Prioritization

Controls are categorized by severity (Critical, High, Medium, Low) to help you focus on the most impactful security issues first. Risk scoring algorithms calculate weighted compliance scores.

Severity Risk Weight Example Controls Remediation Priority
Critical 10x MFA enforcement, Security defaults Immediate
High 7x Modern auth, External forwarding Within 48 hours
Medium 4x DLP policies, Admin consent Within 1 week
Low 1x Calendar sharing, Password expiry Within 30 days

‚ö° Safe Remediation Workflows

Preview changes before applying with -WhatIf mode and approval gates

What-If Preview Mode

Test remediation actions without modifying your tenant. The -WhatIf parameter shows exactly what changes would be made, allowing you to validate fixes before applying them to production. 

# Preview remediation changes (SAFE - no modifications)
powershell -File scripts/powershell/PostRemediateM365CIS.ps1 -WhatIf

# Example output:
# What if: Would disable external calendar sharing
# What if: Would enable admin consent workflow
# What if: Would configure safe attachments policy

# Apply changes after review
powershell -File scripts/powershell/PostRemediateM365CIS.ps1 -Force

Automated Remediation with Approval Gates

GitHub Actions integration supports automated remediation with manual approval gates. Configure which controls can be auto-fixed and require approval for sensitive changes. 

# GitHub Action with approval workflow
- name: Run Remediation
  uses: Heyson315/Easy-Ai@v1
  with:
    enable-auto-remediation: true
    auto-approve-remediation: false  # Requires manual approval
    remediation-controls: '1.1.1,1.1.3,2.1.1'  # Specific controls

# Approval required before changes are applied
# GitHub Environment protection rules enforce approval
  • Environment-based approval workflows
  • Control-specific remediation targeting
  • Detailed remediation logs and audit trails
  • Rollback procedures for failed remediations

ü§ñ GitHub Actions CI/CD

Seamless integration with GitHub workflows for automated auditing

Published GitHub Action

Use our pre-built action (Heyson315/Easy-Ai@v1) in your workflows. No need to clone the repository or manage dependencies - just reference the action and configure your secrets. 

name: Monthly M365 Security Audit

on:
  schedule:
    - cron: '0 2 1 * *'  # 2 AM on 1st of month
  workflow_dispatch:

jobs:
  audit:
    runs-on: ubuntu-latest
    steps:
      - name: Run M365 Audit
        uses: Heyson315/Easy-Ai@v1
        with:
          tenant-id: ${{ secrets.M365_TENANT_ID }}
          client-id: ${{ secrets.M365_CLIENT_ID }}
          client-secret: ${{ secrets.M365_CLIENT_SECRET }}
          spo-admin-url: ${{ secrets.SPO_ADMIN_URL }}
          generate-dashboard: true
          upload-to-security-tab: true

Advanced Outputs & Metrics

Access 25+ output variables including risk scores, compliance trends, severity breakdowns, and SARIF reports for GitHub Security tab integration.

Risk Scoring:
risk-score, critical-findings, high-findings, medium-findings, low-findings
Compliance Trending:
compliance-trend, new-failures, fixed-issues, trend-direction
Security Integration:
sarif-report, security-findings-count
Remediation:
remediated-controls, remediation-report
# Use outputs in subsequent steps
- name: Evaluate Compliance
  run: |
    echo "Risk Score: ${{ steps.audit.outputs.risk-score }}"
    echo "Trend: ${{ steps.audit.outputs.compliance-trend }}"
    
    # Fail if compliance drops below threshold
    if (( $(echo "${{ steps.audit.outputs.compliance-score }} < 80" | bc -l) )); then
      echo "::error::Compliance below 80% threshold!"
      exit 1
    fi

SARIF Security Tab Integration

Automatically upload audit findings to GitHub's Security tab using SARIF 2.1.0 format. Filter by severity threshold and view security issues alongside code scanning results. 

# Enable Security tab upload
- uses: Heyson315/Easy-Ai@v1
  with:
    upload-to-security-tab: true
    security-severity-threshold: high  # Only high/critical

Batch Tenant Audits

Audit multiple M365 tenants in parallel using matrix strategies or batch configurations. Generate per-client reports and consolidated compliance dashboards. 

# Option A: Matrix strategy (parallel execution)
strategy:
  matrix:
    tenant: [Client-A, Client-B, Client-C]
steps:
  - uses: Heyson315/Easy-Ai@v1
    with:
      tenant-id: ${{ secrets[format('TENANT_ID_{0}', matrix.tenant)] }}

# Option B: Batch configuration
- uses: Heyson315/Easy-Ai@v1
  with:
    tenant-config: |
      [
        {"name": "Client-A", "tenantId": "guid-1"},
        {"name": "Client-B", "tenantId": "guid-2"}
      ]
  • Parallel tenant audits (10+ clients simultaneously)
  • Per-client report generation and artifact storage
  • Aggregated compliance metrics across all tenants
  • Client-specific remediation workflows
